# Cortex Agent — Tool System (Full Reference)

> Generated by scanning `src/ai/agent_bridge.py` and `src/agent/src/tools/`
> directly (schemas, dispatch table, and real implementations cross-checked
> against each other — nothing here is guessed). Reflects the codebase as
> of 2026-07-12 / v2.8.0.

---

## 1. How a tool call actually flows

```
LLM emits a tool call
        │
        ▼
Autonomy gate (autonomy_manager) ── ASK mode may block; AUTO mode lets Bash through
        │
        ▼
Safety guard (doom-loop / stale-read / max-iterations checks)
        │
        ▼
Settings gate (Safety & Permissions: allow_file_create / allow_terminal / allow_file_delete)
        │
        ▼
Subscription gate (WebSearch / WebFetch require a Cortex subscription)
        │
        ▼
_TOOL_DISPATCH_MAP lookup  ──►  real tool handler
        │
        ├─ if tool_name starts with "mcp__"  →  routed to the MCP manager instead
        └─ if tool_name is Bash/PowerShell/LS →  bridge-native handler (no dict entry)
```

Every dispatch path returns a `ToolResult(tool_id, result, success, error)`.

## 2. Tools the model can actually call today (14 built-ins)

These are the only entries in `_TOOL_SCHEMAS` — the list that's actually
sent to the LLM as available functions. If it's not in this table, the
model literally cannot call it, no matter what dispatch code exists for it
(see Section 4).

| Tool | Purpose | Key parameters | Notable behavior |
|---|---|---|---|
| **Read** | Read a file — text, images, PDFs, Jupyter notebooks | `file_path`, `offset`, `limit` | Per-model character cap (`file_cap`, logged as `[CTX] Model limits`); **Smart Chunk Boundary Snapping** — when reading a slice of a large file, the end of the chunk snaps to the nearest function/class boundary (regex-based per language: Python, JS/TS, Java, Go, …) instead of cutting mid-function |
| **Write** | Create a new file or overwrite an existing one | `file_path`, `content` | Gated by Settings → Safety & Permissions → "Allow file creation" |
| **Edit** | Exact string replace in a file | `file_path`, `old_string`, `new_string`, `replace_all` | Has a staleness check against the last `Read` of that file (edits fail if the file changed since it was last read) |
| **Bash** | Run a shell/PowerShell command in the project root | `command`, `timeout` (default 30s) | Bridge-native (`BridgeBashTool`) — no real `BashTool.py` exists, this is a Cortex-specific implementation. Gated by "Allow terminal execution" and "Allow file deletion" (for destructive commands); goes through an optional sandbox command allow-list; AUTO mode never fully blocks Bash (it's the fallback when Edit/Write fail) |
| **Glob** | Find files by glob pattern (e.g. `**/*.py`) | `pattern`, `path` | |
| **Grep** | Regex search across file contents | `pattern`, `path`, `glob`, `case_insensitive` | ripgrep-backed (`GrepTool`); pre-warmed at startup so Windows Defender caches trust the binary on first launch |
| **SementicSearch** *(sic — real name has the typo)* | Natural-language / meaning-based code search | `query`, `path`, `top_k` (≤50), `min_similarity`, `output_mode` (ranked/content/files_with_matches), `file_extension`, `force_reindex` | **Subscriber-only.** Falls back to suggesting Grep/Glob for non-subscribers. Backed by the project's semantic embedding index |
| **LS** | List a directory's contents | `path` | Bridge-native (`BridgeLSTool`) — no real agent equivalent exists |
| **TodoWrite** | Maintain the visible todo list | `todos[]` (each: `id`, `content`, `activeForm`, `status`) | Model is instructed to call this first for any 3+ step task; excessive back-to-back calls with no real progress trip a safety warning (`_todo_write_streak`) |
| **AskUserQuestion** | Ask the user a multiple-choice question | `questions[]` (1–4, each with `question`, `header`, `options[]`, optional `multiSelect`) | Renders as an interactive card in chat (this replaced an earlier bug where models sometimes emitted raw `<questions>` JSON as text instead of a real tool call) |
| **WebFetch** | Fetch a URL and return its content as markdown | `url`, `query` | **Subscription-gated** (see Section 3) unless a connected MCP server can do the same job |
| **WebSearch** | Web search — titles, URLs, snippets | `query`, `allowed_domains[]`, `blocked_domains[]` | **Subscription-gated** (see Section 3) unless a connected MCP web-search tool is available |
| **Loop** | Run a *verified* iteration loop — goal + hard gate (tests/lint/build), not just "try again" | `action` (start/verify/status/stop), `loop_id`, `goal`, `verify[]` (name/command/parser/timeout), `allow_paths[]`, `deny_paths[]`, `max_iterations` (default 8), `max_stalls` (default 3), `max_wall_clock_min` (default 30), `approach`, `files_touched[]`, `outcome` | Refuses to start with no command-based verify check available. Detects test tampering (deleted test, weakened assertion, skip marker) and reverts the commit immediately — 3 such attempts halts the loop. A **separate reviewer model** (never the one that edited) audits the full diff before letting the loop stop. Every iteration is git-checkpointed on `cortex/loop/<id>` |
| **LivePreview** | Render + test a local HTML file in Cortex's own embedded panel (2.8.0) | `action` (open/read/console/close), `path` | No external browser. `open` shows the file to the user; `console` returns captured JS console output since the last load; `read` returns the *rendered* (post-JS) visible text; runs on the UI thread via a signal + `threading.Event` handoff with a 15s timeout so a stuck UI can't deadlock the agent |

Plus, dynamically appended at runtime:

| Tool namespace | Purpose |
|---|---|
| **`mcp__<server>__<tool>`** | Every tool exposed by a connected MCP (Model Context Protocol) server — e.g. `mcp__github__create_issue`, `mcp__playwright__browser_click`. Namespaced by server so tools from different servers never collide. **Requires an active Cortex subscription** (`MCPManager._has_active_subscription()` — fails closed: any error while checking = treated as unsubscribed) |

## 3. Subscription gating (fail-closed)

Two independent gates both check for a Cortex subscription:

1. **`_SUBSCRIPTION_REQUIRED_TOOLS = {"WebSearch", "WebFetch"}`** — blocked
   pre-dispatch for non-subscribers, *unless* the user has connected their
   own MCP server that can search/fetch the web (e.g. Tavily) — in that
   case the model is redirected to use the MCP tool instead of being shown
   a paywall card. This logic lives in `_mcp_web_tool_names()`, which scans
   connected MCP tool names for `search`/`fetch`/`crawl`/`browse`/`extract`
   keywords.
2. **MCP tools themselves** (`mcp__*`) — gated separately inside
   `MCPManager`. `get_tool_definitions()` returns an empty list for
   non-subscribers, so the model is never even shown the tools exist;
   `call_tool()` also independently refuses if called anyway. Both checks
   fail **closed** (any exception while checking subscription status = "no
   subscription").
3. **SementicSearch** is subscriber-only too, but fails *open* into a
   suggestion to use Grep/Glob rather than hard-blocking.

## 4. Dispatch handlers that exist but are NOT reachable by the model

Real scan finding, not a guess: `_TOOL_DISPATCH_MAP` (and the dispatch
methods it points to) contains **9 more entries than `_TOOL_SCHEMAS`
advertises**. These have working `async def _dispatch_*` implementations,
but since no schema for them is ever sent to the LLM, the model has no way
to invoke them today:

| Dead-to-the-model tool | Dispatch method | Apparent purpose |
|---|---|---|
| `TaskCreate` | `_dispatch_task_create` | Create a task-graph entry |
| `TaskUpdate` | `_dispatch_task_update` | Update a task's status |
| `TaskList` | `_dispatch_task_list` | List tasks |
| `TaskGet` | `_dispatch_task_get` | Get one task |
| `TaskStop` | `_dispatch_task_stop` | Stop a task |
| `MCP` | `_dispatch_mcp` | Unclear — separate from the `mcp__*` namespaced routing, which is handled elsewhere |
| `TeamCreate` | `_dispatch_team_create` | Create a multi-agent "team" |
| `TeamDelete` | `_dispatch_team_delete` | Delete a multi-agent "team" |
| `PlanBuild` | `_dispatch_plan_build` | Build/execute a saved plan (there's a live reference to `PlanBuild` and `plan_id` elsewhere in the file, suggesting this was wired into *something*, just not the tool-schema list) |

There's also a matching activity-name lookup table (`{"TeamCreate":
"team_create", "TaskCreate": "task_create", ...}`) used for UI/log
formatting, which further confirms these were designed features, not
typos. `src/agent/src/tool_registry.py` — an 8-line stub — is the missing
piece: the code comments explicitly say registry-based tool loading is
"skipped to avoid false warnings" because the registry doesn't provide a
functional `get_all_base_tools()`. That registry is almost certainly where
these 9 tools' schemas were meant to be defined and merged in.

**Net effect:** multi-agent "teams" and a standalone task-graph/plan-build
system exist in the codebase in a half-wired state — real dispatch logic,
no way for the model to trigger it. Worth knowing if you ever wonder why
the agent never "creates a team" or manages tasks outside the `TodoWrite`
list — the plumbing is there, the tap just isn't connected.

There are also several tool **directories** under `src/agent/src/tools/`
with no dispatch entry at all (i.e., not even vestigial-reachable):
`AgentTool`, `EnterPlanModeTool`, `ExitPlanModeTool`, `ListMcpResourcesTool`,
`NotebookEditTool`, `PowerShellTool` *(only used inline as a synonym for
Bash safety-gating, no separate schema/dispatch)*, `REPLTool`,
`SendMessageTool`, `SkillTool`, `SleepTool`, `ToolSearchTool`,
`VisionAgentTool`. These look like scaffolding for future tools (some
names mirror Claude Code's own toolset) rather than anything currently
exercised.

## 5. Safety & permission layers that wrap every tool call

- **Autonomy modes** (`autonomy_manager.AutonomyLevel`) — `ASK` (default)
  requires explicit permission for risky actions; higher autonomy levels
  let more through automatically. Bash is special-cased to never be fully
  blocked in AUTO mode, since it's the agent's fallback when Edit/Write
  fail.
- **Safety guard** — doom-loop detection, max-iteration caps, "read before
  edit" enforcement, stale-read detection, error-recovery loop detection.
  Runs before every dispatch.
- **Settings → Safety & Permissions** — three user-facing toggles gate
  real tool behavior directly: *Allow file creation* (blocks `Write`),
  *Allow terminal execution* (blocks `Bash`/`PowerShell`), *Allow file
  deletion* (blocks destructive Bash commands specifically).
- **Sandbox manager** (optional) — an allow-list for shell commands,
  checked before `Bash` executes if enabled.
